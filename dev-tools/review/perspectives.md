# AIレビュー観点（汎用版）

本観点は、AIが「一次フィルター」として技術的整合性を網羅的にチェックし、人間が「ビジネス価値」の判断に集中できるように設計されています。

---

## 観点1: 機能の正確性とビジネスロジック 🔴
Issueの「受入条件（Acceptance Criteria）」との整合性をチェックします。
- **要件充足:** Issueで定義された受入条件を全て満たしているか。
- **エッジケース:** 境界値、異常系、ヌル値等での動作が考慮されているか。
- **エラーハンドリング:** 例外時にシステムがクラッシュせず、適切に回復するか。

## 観点2: 不適切な変更 🔴
品質を低下させる変更や、問題を隠蔽する変更をチェックします。
- **不適切な抑制:** `@ts-ignore` / `eslint-disable` / 型アサーション `as` による強引なエラー回避。
- **テストの改ざん:** テストのアサーションをバグに合わせて変更していないか。
- **デバッグコード:** `console.log`、`debug: true` 等の混入。
- **認可チェック:** 認証・認可ロジックの削除や弱体化。
- **放置コード:** コメントアウトされたコード、テストの `skip` / `.only` の放置。

## 観点3: セキュリティ 🔴
技術的な脆弱性と、プロジェクト固有の認可ルールをチェックします。
- **一般的脆弱性:** SQLインジェクション、XSS、CSRF、未検証の外部入力。
- **機密情報:** 環境変数やシークレットのハードコーディング。
- **認可チェック:** サーバー側での権限検証が適切か。

## 観点4: 設計・アーキテクチャ 🟠
プロジェクト全体の整合性と保守性をチェックします。
- **責務の分離:** クラスや関数が単一の役割に限定されているか（SRP）。
- **既存パターンの遵守:** プロジェクト独自のアーキテクチャに従っているか。
- **既存パターンの再利用:** 共通ユーティリティや既存コンポーネントで代用できる実装を独自に作り直していないか。
- **破壊的変更:** 公開APIの型や引数の変更が、既存コードと矛盾していないか。

## 観点5: パフォーマンス 🟠
リソース消費と処理効率をチェックします。
- **クエリ最適化:** N+1問題の検知、適切なインデックス利用。
- **リソース管理:** メモリ、接続の解放、非効率なループ排除。
- **通信効率:** ループ内でのAPI呼び出し、不要なデータフェッチ。

## 観点6: テストと品質保証 🟡
Issueの「受入条件」を検証する仕組みをチェックします。
- **テストの網羅性:** 新機能に対する正常系・異常系テストの有無。
- **テストの信頼性:** テストが環境や順序に依存していないか。
- **重要フローのカバー:** 重要なユーザーフローのE2Eテストがあるか。

## 観点7: 可読性と保守性 🟢
チームの一貫性と長期的な修正のしやすさをチェックします。
- **命名規約:** プロジェクトの命名規則に従っているか。
- **意図の明示:** 「何（What）」ではなく「なぜ（Why）」をコメントしているか。
- **コードの簡潔さ:** 条件分岐のネスト排除（早期リターンの活用）。
