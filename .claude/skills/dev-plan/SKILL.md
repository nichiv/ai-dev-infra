# Feature Plan - 機能実装計画

Issue番号を指定して、既存のドキュメント・コードを調査し、要件整理・設計・実装計画・テスト計画を作成します。

**計画が承認されるまでコードは一切書かない。**

---

## 使用方法

```
/dev-plan <issue_number>
```

引数: $ARGUMENTS

---

## Phase 1: 情報収集

### 1.0 ステータス更新（In Progress）

Issue のステータスを `In Progress` に変更する（プロジェクトの運用に合わせて）。

### 1.1 Issue情報の取得

```bash
gh issue view <issue_number>
```

Issue本文から以下を抽出:
- 機能概要、やることチェックリスト、参照ドキュメント
- `## Dependencies` セクション（依存先の issue 番号一覧）

### 1.2 仕様書の調査

1. **Issue本文に記載された参照ドキュメント** — 最も重要
2. **プロジェクトの設計ドキュメント** — アーキテクチャ、設計原則、開発ガイド等
3. **CLAUDE.md / DEVELOPMENT.md** — プロジェクト固有のルール・規約

### 1.3 既存コードの調査

プロジェクト構成に応じて、以下を調査:
- **DBスキーマ** — マイグレーション定義、モデル定義
- **ビジネスロジック** — サービス層、アクション、API
- **UI** — フロントエンドコンポーネント
- **テスト** — 既存テストの構成・パターン

---

## Phase 2: 要件整理

- **入力:** ユーザーからの入力データ
- **出力:** 期待される結果（画面表示、DB変更等）
- **ビジネスルール:** バリデーション、認可、制約条件
- **非機能要件:** パフォーマンス、セキュリティ、アクセシビリティ

---

## Phase 3: 設計

プロジェクトのアーキテクチャに従って設計する。設計の成果物:
- DBスキーマ設計（プロジェクトの命名規約に従う）
- 各レイヤーの設計（プロジェクトのレイヤードアーキテクチャに従う）
- UI設計
- アクセス制御

---

## Phase 4: テスト計画

- **正常系テスト** — 各機能の基本動作
- **異常系テスト（バリデーション）** — 不正入力の拒否（必須）
- **異常系テスト（認可）** — 未認証/権限なしでの拒否（必須）
- **E2Eテスト** — ユーザーフロー単位（コンテンツ検証含む）

---

## Phase 5: 計画ファイルの作成

`./tmp/plan-<issue_number>.md` に以下の構成で保存:

```markdown
# Issue #<number> 実装計画: <タイトル>

## 1. 概要
## 2. 要件整理（機能要件 / 非機能要件）
## 3. 設計（DB / 各レイヤー / UI / アクセス制御）
## 4. Dependencies（issue body の依存先 issue 一覧。なければ省略）
## 5. 実装手順（依存関係を考慮した順序）
## 6. テスト計画（ユニット / 統合 / E2E）
## 7. ユーザー確認事項（不明点は勝手に判断しない）
## 8. 影響範囲（変更ファイル一覧 / 更新ドキュメント）
```

---

### 5.1 ユーザー確認事項の質問

計画ファイルの「## 7. ユーザー確認事項」がある場合は、**必ず `AskUserQuestion` ツールを使用して質問すること。テキストでの質問は禁止。**

ユーザー確認事項がない場合でも、計画の概要を `AskUserQuestion` で提示して承認を得ること。

回答を計画ファイルに反映してから Phase 6 に進む。

---

## Phase 6: ユーザー承認

計画ファイルを作成したら、概要サマリーと確認事項を提示して承認を求める。

**承認されたら:**

1. 計画内容を GitHub Issue にコメントとして投稿する（意思決定ログとして残す）:

```bash
gh issue comment <issue_number> --body-file ./tmp/plan-<issue_number>.md
```

2. Issue body を更新する:

   a. `gh issue view <issue_number> --json body --jq .body` で現在の body を取得
   b. 以下のセクションを組み立てて `/tmp/issue-body-<issue_number>.md` に保存:
      - **概要**: 計画の内容に沿って修正
      - **Dependencies**: 既存のまま維持（あれば）
      - **対応**: 計画の設計から機能単位でどのような追加や変更を行うか記載
      - **やること**: 計画の「実装手順」からフェーズ別子セクション付きチェックリストに変換
      - **計画書**: 投稿した計画コメントの URL を記載
   c. `gh issue edit <issue_number> --body-file /tmp/issue-body-<issue_number>.md` で更新

   **Issue body の目標構造:**

   ```markdown
   ## 概要
   （計画に基づいて更新された概要）

   ## Dependencies
   - #42

   ## 対応
   - **機能A**: 〜を追加
   - **機能B**: 〜を変更

   ## やること
   ### Phase 1: DB・スキーマ
   - [x] テーブル追加
   - [ ] マイグレーション

   ### Phase 2: API実装
   - [ ] Repository作成
   - [ ] Service作成

   ## 計画書
   - [実装計画](コメントURL) — YYYY-MM-DD
   ```

3. 「計画の投稿と Issue の更新が完了しました。実装に進む場合は `/dev-execute <issue_number>` を実行してください。」とユーザーに案内する。**自動で `/dev-execute` を実行してはならない。**

---

## 重要ルール

- **勝手に判断しない:** 不明点は「ユーザー確認事項」に記載
- **AskUserQuestion 必須:** 不明点やユーザー確認事項はテキストではなく `AskUserQuestion` ツールで質問する
- **コードを書かない:** このスキルは計画のみ
- **仕様書を正とする:** 独自の解釈ではなくドキュメントの記載に基づく
- **既存パターンに従う:** 新しいパターンを導入しない
