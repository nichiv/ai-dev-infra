# Feature Execute - 計画に基づく実装

`/dev-plan` で作成・承認された計画ファイルに基づいて実装を行います。

---

## 使用方法

```
/dev-execute <issue_number>
```

引数: $ARGUMENTS

---

## Step 1: 計画読み込み

`./tmp/plan-<issue_number>.md` を読み、実装手順・テスト計画・影響範囲を確認する。

## Step 2: ブランチ確認

```bash
git branch --show-current
# → <branch_prefix>-<issue_number> であること。異なる場合はユーザーに通知して停止。
```

ブランチ命名規約はプロジェクトの CLAUDE.md または `.config/project.yml` の `branch_prefix` を参照する。

## Step 3: ステータス更新（In Progress）

Issue のステータスを `In Progress` に変更する（プロジェクトの運用に合わせて）。

## Step 4: 実装

計画の「実装手順」に従い、記載された順序で実装する。

**フェーズ完了時の Issue body 更新:**

計画の実装手順がフェーズ分けされている場合、各フェーズの完了時に issue body の「やること」チェックリストを更新する:

1. `gh issue view <issue_number> --json body --jq .body` で現在の body を取得
2. 完了したタスクのチェックボックスを `- [x]` に更新
3. `/tmp/issue-body-<issue_number>.md` に保存
4. `gh issue edit <issue_number> --body-file /tmp/issue-body-<issue_number>.md` で更新

**実装時の参照:**
- プロジェクトの設計ドキュメント（コーディング規約、レイヤー設計等）

## Step 5: テスト実行

プロジェクトのテストコマンドを実行する（例: `make test`, `npm test` 等）。

CLAUDE.md に記載されたテストコマンドに従うこと。

## Step 6: テスト結果検証（ゲート）

以下の条件を **すべて** 確認する。**1つでも満たさない場合は Step 7 以降に絶対に進まない。**

| 条件 | 確認方法 |
|------|----------|
| 全テストが通過 | 終了コード 0 |
| テストが実際に実行された | "0 tests found" や接続エラーではないこと |

**いずれかが NO の場合:**
1. テストログを確認
2. 修正可能 → 修正して Step 5 に戻る
3. 自力で解決できない → **ユーザーに報告して停止。コミットしない。**

## Step 7: Lint

プロジェクトの lint コマンドを実行する（例: `make lint`, `npm run lint` 等）。

エラーがあれば修正。

## Step 8: 再テスト（lint修正後）

lint修正でコードが変わった場合、Step 5-6 を再実行する。

## Step 9: コミット・PR作成

コミットメッセージは Conventional Commits に従う。PR を作成する。

push 時に pre-push フック（Lefthook 等）で自動レビューが実行される場合:

| 結果 | 動作 |
|------|------|
| LGTM（exit 0） | push が成功 → PR作成に進む |
| 指摘あり（exit 1） | push が中断 → 指摘を修正して再コミット・再 push |

## Step 10: ステータス更新（In Review）

Issue のステータスを `In Review` に変更する（プロジェクトの運用に合わせて）。

## Step 11: 完了報告

ユーザーに以下を報告:
- PR URL
- 実装内容のサマリー
- テスト結果
- 計画からの変更点（あれば）

---

## 重要ルール

- **計画に忠実:** 計画に書かれていない機能を追加しない
- **計画の変更が必要な場合:** ユーザーに相談してから変更
- **全テスト通過が必須:** テストが通らないままコミット・PR作成しない
- **PRマージは絶対にしない:** PR作成まで。マージはユーザーが行う

---

## 禁止される判断パターン

1. **「環境の問題だからスキップ」** → 禁止。環境を修正するか、ユーザーに報告して停止。
2. **「テストは通るはずだからスキップ」** → 禁止。実際に通過を確認する。
3. **「前回通ったから今回はスキップ」** → 禁止。毎回実行する。
4. **「このテストは今回の変更に関係ない」** → 禁止。全テスト実行が必須。
5. **テスト未実行を「通過」とみなすこと** → 禁止。

---

## エラー発生時

- **テスト失敗:** → テストログを確認、原因特定して修正
- **計画との不整合:** ユーザーに相談。勝手に計画を変更しない
